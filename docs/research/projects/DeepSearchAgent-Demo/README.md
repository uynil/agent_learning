# DeepSearchAgent-Demo

## Project Summary

`DeepSearchAgent-Demo` is a no-framework deep search agent that turns one user query into a sectioned research report by combining report-structure planning, Tavily search, per-section summaries, reflection loops, and final Markdown formatting.

## Quick Classification

- Style: workflow-heavy, tool-using, single-agent research pipeline
- Best fit: sectioned planner/executor with reflection
- Deep research behavior: present, but implemented as per-paragraph loops rather than a broad standalone planner

## Observed Facts

- `DeepSearchAgent.research()` runs a fixed three-stage flow: generate report structure, process paragraphs, generate the final report.
- `ReportStructureNode` uses `SYSTEM_PROMPT_REPORT_STRUCTURE` and returns a list of up to five paragraph skeletons.
- Each paragraph goes through an initial search, an initial summary, a reflection loop, and a reflection summary.
- The reflection loop is bounded by `Config.max_reflections`.
- `State` persists query, report title, paragraphs, final report, timestamps, and per-paragraph research history.
- `Research` tracks `search_history`, `latest_summary`, `reflection_iteration`, and completion state.
- `Search` records query, URL, title, content, score, and timestamp.
- All prompt nodes use `BaseLLM.invoke(system_prompt, user_prompt)` and expect JSON-shaped outputs.
- The code cleans model output with `remove_reasoning_from_output`, `clean_json_tags`, and `extract_clean_response`.
- Final Markdown formatting has a fallback `format_report_manually()` path if the LLM formatting step fails.
- The agent can save and load full state snapshots as JSON files.
- `Config.max_paragraphs` exists, but the current runtime flow does not appear to enforce it directly.

## Inferences

- This is a single-agent orchestration pipeline, not a multi-agent collaboration system.
- The most important working memory is the paragraph-level `latest_summary`, not a global conversation transcript.
- The prompt design relies on strict JSON contracts plus post-processing instead of open-ended natural language generation.
- The workflow is more like sectioned deep research than free-form chat: the system decomposes a query into report sections and then deepens each section separately.
- Resumability exists at the state layer, but not as an explicit restart workflow in the main UI or agent loop.

## Open Questions

- How often do the JSON-cleaning heuristics have to rescue malformed outputs in practice?
- Does the reflection loop materially improve coverage, or does it mostly re-run the same search shape?
- Should `max_paragraphs` be enforced in code, not just suggested in prompts and config?
- Are the section boundaries generated by `ReportStructureNode` stable enough across very different queries?
- Would a more explicit resume API make this project easier to continue after interruption?
- After dependencies and API keys are configured, does the live run still match the current static call graph?
